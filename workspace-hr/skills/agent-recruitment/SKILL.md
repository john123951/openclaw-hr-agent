# Agent 招聘技能

## 触发条件

当用户表达任何以下意图时触发：
- "我需要一个..."、"帮我找个..."、"招一个..."
- "能不能有个 agent 帮我..."
- "我想要一个助手来..."
- 任何关于新增 agent、新员工、新助手的意图

## 三段式对话流程

### 第一段：意图捕捉（1 轮）

1. 从用户的话中提取：**做什么** + **为什么**
2. **⚠️ 必须先检查本地可用模型**：
   - 运行命令：`openclaw models`
   - 仔细查阅输出中的 Configured models 和 Fallbacks 列表，这些才是当前系统真正配置并健康可用的模型。
   - **绝不**推荐未出现在 `openclaw models` 列表中的模型（即使任何其他文档里推荐过默认模型）。
3. 根据你的专业判断，结合 `openclaw models` 的输出，推断出以下信息：
   - 推荐的模型（从 Configured models 列表中挑选。轻量任务选基础模型，复杂任务选最强推理模型。如果遇到限流退回，请主动给出列表里的第二备选模型让用户抉择）
   - 推荐的 agent ID（英文小写，用连字符分隔）
   - 推荐的 agent 名称（含 emoji）
   - 推荐的工作时间/频率
   - 推荐的工具权限（参考岗位模板）
3. 展示你的初步方案
4. **只问一个**你无法推断的核心问题

### 第二段：补全缺失（0-2 轮）

只在以下信息缺失时才追问（每轮只问 1 个）：
- 具体参数（如监控哪些股票、查什么数据）
- 通信渠道选择（提供选项给用户，如 A.飞书定群 B.Telegram C.仅后台待命）
  - **⚠️ 绝对红线纪律**：无论用户当前正通过哪个渠道/群组和你聊天，**绝不准**默认将新员工绑定到当下的聊天群组中（因为当前群组是你的 HR 专属办公地）。
  - 如果用户选择绑定并推送到任意外部群聊（飞书、Telegram、Discord等），你**必须明确追问**该渠道的目标群组 ID（如飞书的 `oc_xxx` 格式，Telegram 的 `-100xxx` 格式），除非用户已在前序对话中自己发出了 ID。
**⚠️ 关键：飞书群聊体验设置（仅由 HR 内部决策，绝不询问用户）**
为了给老板极致简洁的体验，基础框架配置（如打字机流式输出、引用模式、群组免打扰）由你**默认决定**，不要拿这些技术细节去烦用户：
- **默认原则**：除非用户强烈主动要求“必须要@才回复”，否则**所有岗位的默认行为都是**“免 @ 全局监控（`--require-mention false`）”！
- 为了配合“免 @”全局监控且保持群内回复可读性，**强烈建议同时开启**引用回复（`--reply-to all`）。

**规则**：
- 用选择题（A/B/C）代替开放题
- 如果第一段信息已充足，跳过直接到第三段

```
[Emoji] [Agent 名称]
─────────────────
[职责一句话]
模型：[model]
渠道：[channel info]
[如有特殊配置，列出]

说"好"我就开始！
```

用户确认后，调用 `agent-provisioning` skill 开始创建。记得把你自动决策的群聊体验选项（如 `--require-mention false`、`--reply-to all`）带入给 Provisioning。

## 岗位模板参考

读取 `{baseDir}/../../templates/jobs/job-profiles.json` 获取预设岗位模板。
模板包含：推荐模型、默认工具权限、推荐 skills、知识库预设内容。

如果用户需求不匹配任何模板，使用最小权限安全默认值：
- allow: `["exec", "read", "write", "edit", "web_fetch", "sessions_list", "sessions_send", "sessions_history", "memory_search", "memory_get", "message"]`
- deny: `["browser", "canvas", "nodes", "sessions_spawn"]`

> ⚠️ **基线权限红线**：`write`/`edit`（知识沉淀）、`sessions_list`/`sessions_send`/`sessions_history`（内部求助）、`web_fetch`（查阅资料）是所有员工的"生命线"权限。**任何岗位都必须包含这些权限**，否则员工在遇到问题时将无法求助，也无法维护知识库。

## 常见岗位速查（仅作能力参考）

| 关键词 | 岗位 | 模型选择策略 |
|-------|------|------|
| 天气、气温、下雨 | 天气助手 | 优先选轻量基础模型（如 gpt-3.5/haiku 级别） |
| 股票、盯盘、行情、K线 | 股票监控 | 优先选基础或中等模型 |
| 写文章、文案、内容 | 文案创作 | 优先选中等模型（如 gpt-4o-mini/sonnet 级别） |
| 代码、编程、开发 | 程序员 | 必须选最强模型（如 gpt-4o/opus 级别） |
| 运营、推广、社交 | 运营专员 | 优先选中等以上模型 |
| 调研、报告、分析 | 研究员 | 必须选强推理模型 |
| 产品、需求、PRD | 产品经理 | 优先选中等以上模型 |
| 管理、CEO、战略 | CEO 顾问 | 必须选最强模型 |
